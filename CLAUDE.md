# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

AgentFlow ‚Äî a self-hosted platform for writing/running LangGraph/LangChain Python scripts in the browser. FastAPI backend + Next.js frontend (static export, served by FastAPI). Each user script gets its own isolated venv.

**Backend tests** live in `backend/tests/` (pytest). Most are fast unit tests over the stable, security-critical logic (auth crypto, filename path-safety, subprocess env scrubbing, schema validation, execution-retention) that run against a **throwaway temp sqlite DB + data dir** wired up in `tests/conftest.py` ‚Äî no server, no real `backend/data`. `tests/test_sdk_contract.py` locks the public `agentflow` surface + key kwargs scripts rely on. The langchain/langgraph stack is imported *lazily* inside `agentflow.get_llm()`/`get_agent()`, so importing `agentflow` in a test is cheap. **`tests/test_execution_engine_run.py` is the home for "I ran a script and it broke" regressions**: it drives the real engine end-to-end (writes a `main.py`, calls `start_execution()`, spawns the runner subprocess, asserts on the persisted `Execution` row + logs). A test script has no per-script venv, so the engine falls back to the backend python (the test interpreter) ‚Äî an import error / crash reproduces **naturally without building a venv** (fast, hermetic). The seed case is a **missing dependency** (imports an uninstalled package ‚Üí `failed` + `ModuleNotFoundError` in `execution.error` + a persisted error log). **The suite grows by regression:** when a script hits a bug, add a test reproducing it ‚Äî runtime/execution bugs go in `test_execution_engine_run.py` (copy its `_run(db, main_py, ‚Ä¶)` helper), pure helpers get a plain unit test, DB-touching logic takes the `db` fixture. We deliberately **don't** statically test the `examples/*.py` files (that only catches "the example broke", not real runtime behavior). See `backend/tests/README.md` for the how-to. There is still **no frontend test suite** ‚Äî verify UI flows by running the app.

## Architecture you must know

### Two Python runtimes ‚Äî keep them straight

1. **Backend Python** in `backend/.venv` runs FastAPI/SQLAlchemy/APScheduler.
2. **User script Python** in `backend/data/scripts/<id>/.venv` runs whatever the user wrote.

The split is *platform code vs. untrusted user code*: user scripts get an isolated per-script venv; platform code does not. The **one deliberate exception is the built-in "AI ËÑöÊú¨Âä©Êâã" assistant** (`services/assistant_seed.py`). It is platform code that happens to run *through* the user-script execution pipeline (subprocess + runner + tracer + WS, so its streaming answer / tool-trace render for free) ‚Äî but because it's **trusted platform code with fixed deps**, it runs on the **backend python** (`sys.executable`) and reuses the platform's own langchain stack, so it has **no per-script venv**. `execution_engine` selects `sys.executable` whenever `script.name == ASSISTANT_SCRIPT_NAME` (never a script venv), and `seed_assistant` deletes any stale assistant venv a previous build created. Net effect: upgrading the platform upgrades the assistant ‚Äî nothing per-script to build or keep updated. This is why the langchain/langgraph/deepagents stack lives in `backend/requirements.txt` **as well as** `venv_manager.BASELINE_PACKAGES`; **keep the two lists in sync**.

`backend/agentflow/__init__.py` is imported by user scripts, NOT by backend (except by the assistant subprocess, which runs on the backend python but still adds `BACKEND_ROOT` to `sys.path` via the runner). The runner adds `BACKEND_ROOT` to `sys.path` so user-script venvs can `from agentflow import ‚Ä¶` without installing it. Don't add agentflow to `requirements.txt`.

### Subprocess plumbing (the part with sharp edges)

`services/execution_engine.py` and `services/venv_manager.py` deliberately avoid `asyncio.create_subprocess_exec`. Use sync `subprocess.Popen` + background thread + `asyncio.Queue` + `loop.call_soon_threadsafe`. Reasons (do not "fix" them):

- Windows `SelectorEventLoop` (debugpy injects this) raises `NotImplementedError` on `asyncio.subprocess_exec`.
- Pip progress bars emit `\r` (no `\n`) ‚Äî we hand-split on both.
- `_clean_env()` in `venv_manager.py` strips `PYTHONPATH` / `PYTHONHOME` / `VIRTUAL_ENV` / `PYDEVD_*` / `DEBUGPY_*` / `PYCHARM_*`. **Never replace with `os.environ.copy()`** ‚Äî debugpy infects user venvs that don't have it installed.
- User-script subprocesses use `CREATE_NEW_PROCESS_GROUP` on Windows so CTRL_C from `uvicorn --reload` doesn't kill them.
- `launch.json` sets `"subProcess": false` because debugpy 2026+ monkey-patches `subprocess.Popen` to inject pydevd into child pythons. Don't re-enable it.

#### Execution sandbox (defense-in-depth for user-script runs) ‚Äî POSIX only

Both spawn sites (the one-shot `Popen` in `execution_engine.py` **and** the warm-worker `Popen` in `worker_pool.py`) apply two independent, best-effort layers via **one pair of helpers in `venv_manager.py`** ‚Äî reused so the two paths never drift. **Both no-op on Windows and degrade gracefully; a run is never failed just because a layer couldn't apply** (mirrors the existing `_sandbox.py` / `CREATE_NEW_PROCESS_GROUP` degrade philosophy). This is distinct from the **opt-in** `agentflow/_sandbox.py` (`run_bash`/`run_python`), which has a *different, stricter* threat model (env-scrubbed, empty cwd) because it runs agent-generated code; the main path is *semi-trusted* and must keep the script's own secrets / LLM creds / network.

- **`make_run_preexec()` ‚Äî rlimits (`preexec_fn`).** Caps address space (`RLIMIT_AS`, default 4 GiB, `AGENTFLOW_RUN_MEM_MB`) so one script can't OOM the host, plus `RLIMIT_CORE=0`. `RLIMIT_FSIZE` (`AGENTFLOW_RUN_FSIZE_MB`) and `RLIMIT_NPROC` (`AGENTFLOW_RUN_NPROC`) are **off by default** (`0`) ‚Äî NPROC is per-UID (counts the whole backend's processes), a real footgun, so it's opt-in. **Deliberately NO `RLIMIT_CPU`** (runs are network/LLM-bound; the wall-clock `EXECUTION_TIMEOUT` bounds runaways, and a warm worker would accumulate CPU across jobs and self-kill) and **NO `os.setsid()`** (keeps process-group / cancellation semantics unchanged). Returns `None` where `resource` is absent ‚Üí Popen just gets no `preexec_fn`.
- **`maybe_wrap_sandbox()` / `build_bwrap_prefix()` ‚Äî filesystem jail (bubblewrap).** Wraps the child argv so it runs in a mount+pid namespace containing only: system dirs (ro), the **`agentflow` SDK subdir of `BACKEND_ROOT`** (ro ‚Äî never the whole tree, so `data/.secret_key` / the DB / other scripts stay outside the namespace = invisible), and **this script's own dir** (rw; per-job `run_dir`s live under it, so the worker jails to `script_dir`). Network + env are **not** unshared (LLM/MCP + the script's own secrets keep working). Gated by `AGENTFLOW_SANDBOX` (`auto` default / `off`); `auto` runs a **one-time functional probe** (`_bwrap_probe`) and disables itself where bwrap is missing or the kernel blocks unprivileged namespaces. The **built-in assistant is excluded** (`is_assistant` ‚Üí bare command) ‚Äî it's platform code that legitimately reaches backend assets / the loopback gateway. On an `OSError` at launch the caller falls back to the bare command (`is`-compare vs `base_cmd`). Extra binds for shared mounts: `AGENTFLOW_SANDBOX_RO_BINDS` / `AGENTFLOW_SANDBOX_RW_BINDS`. The Docker image installs `bubblewrap`; CI installs it too so `tests/test_venv_manager.py::test_bwrap_hides_out_of_jail_files` actually exercises the jail (else it skips).

### Runner protocol

`_write_runner()` in `execution_engine.py` generates a tiny `_runner_<execution_id>.py` that wraps the user's entry function. User code communicates with the platform via lines prefixed with `__AGENTFLOW__<json>` on stdout. Anything else is captured as `raw` / `error` log level.

**cwd & where the user's files live**: the subprocess runs with `cwd = run_dir` (`script_dir/runs/<execution_id>/`, per-execution & isolated). The user's own files (from `script.files`) are written to **both** `script_dir` (for `import`/venv/persistence ‚Äî the runner adds `script_dir` to `sys.path` and imports `main.py` from there) **and** `run_dir` (so intuitive relative reads like `open("data.txt")` just work against the cwd). The write to the **shared** `script_dir` **skips files whose on-disk content is already identical** ‚Äî otherwise concurrent runs of the same script race (one truncates+rewrites `main.py` while another run's subprocess is importing it ‚Üí *"module user_script has no attribute run"*); `tests/test_execution_engine_run.py::test_unchanged_script_file_not_rewritten` guards it. The runner's `_emit` uses `json.dumps(..., default=str)` so a `run()` return value with a **nested** non-JSON-native object stringifies best-effort instead of crashing the runner with a raw `TypeError` (the top-level result guard only stringifies a non-container top level). run_dir is ephemeral per run, so anything the script writes stays out of the source tree; runtime files (`_runner.py`/`_input.json`/`skills/`) are written after the user files and win on any name clash. Paths are also exposed via env: `AGENTFLOW_RUN_DIR` (cwd), `AGENTFLOW_WORKSPACE_DIR` (persistent shared `script_dir/workspace`), `AGENTFLOW_SCRIPT_DIR`.

The runner is **fully async** (`asyncio.run(_main())`). `nest_asyncio` is applied at startup so sync `agent.invoke()` can call `asyncio.get_event_loop().run_until_complete()` internally (needed for async MCP tools wrapped by `_ensure_sync()` in `agentflow.get_tools`). Third-party loggers (`mcp`, `httpx`, `openai`, etc.) are silenced to `WARNING` to prevent INFO lines appearing as `[ERR]` in the log panel.

MCP servers are connected via `MultiServerMCPClient` (no async context manager ‚Äî removed in `langchain-mcp-adapters` 0.1.0; use `client = MultiServerMCPClient(...); tools = await client.get_tools()` instead). Injected tools are stored in `agentflow._injected_tools` before user code runs. Async-only MCP tools (`StructuredTool` with `coroutine` but no `func`) are wrapped by `_ensure_sync()` in `agentflow.get_tools()` so sync `agent.invoke()` works.

### Script input schema (typed `run()` input)

A script can declare a **JSON Schema contract** for its `run(input)` ‚Äî turning the untyped `dict` into a validated interface. The **source of truth is the script itself**: a module-level `INPUT_SCHEMA` (a JSON-Schema dict, or `MyModel.model_json_schema()`). `Script.input_schema` (JSON column, Alembic **`0008`**) is just a **cache** of it. When present it drives three things: (a) **pre-run validation** (bad input ‚Üí 422 / a clean `failed` run, never reaching user code), (b) a **typed /docs call example**, (c) an **auto-rendered form** on the run page.

- **Extraction** (`services/script_schema.py`) is two-level, cheapest first: `_static_extract` resolves a literal `INPUT_SCHEMA = {...}` via `ast.literal_eval` (0 cost, no venv, no code run); if the name is assigned but *computed* (Pydantic), `_introspect` writes the files to a throwaway dir and imports the module in the script's venv (or backend python) to dump the resolved schema. Cheap in practice because `import agentflow` does **not** pull langchain (lazy). `refresh_script_schema(db, script)` persists the result; `compute_schema` is the pure resolver; `validate_input(schema, data)` raises `ValueError` on mismatch (and is a **no-op on a broken/absent schema** ‚Äî we only ever block on bad *input*, never a bad *schema*).
- **Triggers**: `PUT /scripts/{id}/files` refreshes it in a `BackgroundTasks` (static path is instant; the subprocess path stays off the request); `POST /scripts/{id}/schema/sync` forces a synchronous re-derive; the MCP gateway `write_script_file` refreshes + returns it and there's a `sync_script_schema` tool. **Validation** runs at the API layer (`routers/executions.py::_validate_input_or_422` on `create_execution`/`run_sync`) *and* universally inside `start_execution` (so eval/cron/rerun get a clean `failed` run). `jsonschema` is a pinned backend dep (`requirements.txt`).
- **Frontend**: `lib/schemaForm.ts` (`exampleForSchema` / `isFormRenderable` / `fieldKind`) + `components/SchemaForm.tsx` (per-field widgets: string/number/boolean/enum/string-array, nested ‚Üí raw-JSON field) + `components/SchemaInput.tsx` (Form‚ÜîJSON toggle wrapping the classic `InputPresetEditor`, wired into the run page). The `/docs` page builds the typed `input_data` example from the selected script's schema. **Chat scripts keep the fixed `{message, history}` shape ‚Äî don't add `INPUT_SCHEMA` there.** The scripting guide (`assets/skills/agentflow-scripting/SKILL.md`) + the assistant prompt document the convention.

### Warm worker pool (opt-in, serverless-style reuse)

Every normal run spawns a fresh `python _runner.py` that re-imports the whole langchain/langgraph stack (the profiler's dominant `cold_import`). A **per-script persistent worker** (`services/worker_pool.py`) keeps one interpreter alive between runs so run #2+ skips that import; `keep_warm` preheats the heavy stack so run #1 is warm too.

- **GATED OFF by default**: the whole pool is a no-op unless **`AGENTFLOW_WARM_WORKERS`** is enabled *and* the script has `warm=True` (default True). `Script.warm` / `Script.keep_warm` (Booleans, Alembic **`0009`**). `warm=False` opts a script back into the classic fresh-process isolation. With the flag off the engine path is byte-for-byte the old one-shot.
- **Design**: one long-lived subprocess per script, booting once (imports `agentflow` + the user's `main.py`), then reading **one job per line from stdin** and streaming the same `__AGENTFLOW__` events on stdout. Serial per worker (an `asyncio.Lock`); a `job_done` sentinel frames each job. **Per-job config crosses via the job payload `env`** (LLM creds / secrets / MCP / skills / run_dir) ‚Äî *nothing baked to disk*, so MOST config changes (MCP/skills/secrets) need **no restart**; only a **code edit** (`invalidate_worker`, wired into the file PUT/DELETE + gateway `write_script_file` + `delete_script`) or a **venv change** (python path differs) retires a worker. Idle workers are reaped after `AGENTFLOW_WORKER_IDLE_TTL` (default 600s); `AGENTFLOW_WORKER_MAX_JOBS` recycles after N jobs. Between jobs the worker resets `_af._injected_tools` (to **`[]`, never `None`** ‚Äî a no-MCP job leaves it as-is and `get_tools()`/`get_agent()` iterate it, so `None` would crash a bound-to-nothing agent) + the tracer usage tally (`_tracer.reset_usage`) and clears the job's env keys, but **module globals persist** (the documented serverless tradeoff). Because `agentflow` is imported **once at boot** (before any job's path env exists) and reused, anything derived from per-run env must resolve **lazily** per access, not be snapshotted at import ‚Äî `agentflow.paths` (`run_dir`/`workspace`/`script_dir`) is a property-backed object for exactly this reason (a snapshot would freeze every path to the boot cwd = script dir for all reused jobs, silently misdirecting `paths.workspace` writes and `image()`/artifacts). The per-job runner catches **`SystemExit`** (a user `sys.exit()` / `raise SystemExit`) and fails only that job ‚Äî it is a `BaseException`, so a plain `except Exception` misses it and it would escape `run_until_complete`, print asyncio *"Task exception was never retrieved"* noise into the run log, and kill the interpreter (a needless cold reboot). `os._exit()` still hard-kills the worker; the pool self-heals on the next acquire. (`tests/test_worker_pool.py::test_sys_exit_does_not_kill_worker`.)
- **Engine integration**: `start_execution` shares `_handle_event_line` (event ‚Üí DB/WS) and `_finalize_run` (status/usage/retry/prune) between the one-shot and worker paths; `_run_via_worker` acquires a worker (`worker_pool.manager`), sends the job, drains through the shared handler, and finalizes with `returncode=None`. **`acquire()` is serialized per-script** (a `WorkerManager._locks[script_id]` `asyncio.Lock`): without it a cold-boot stampede (eval runs its cases concurrently; concurrent API runs of the same script) had each caller see the others' still-booting worker as "not alive", kill it mid-boot, and spawn its own ‚Äî so all but the last raised *"worker exited during boot"*. The first caller now boots while the rest await and reuse the warm worker; per-job execution stays serialized by the separate per-worker `run_job` lock. Cancellation registers `worker.proc` under `_procs[execution_id]` so `stop_execution` kills it (then the worker is invalidated). Timeout / crash retires the worker so the next run is clean. **Cancel correctness (both paths):** `stop_execution` records the intent in `_cancelled_ids` **before** the `_procs` lookup ‚Äî the worker path only registers `_procs[execution_id]` *after* the (possibly slow) `acquire()`/boot, so a stop clicked in that window would otherwise return early without marking anything and `_finalize_run` would overwrite the row's `cancelled` back to completed/failed while the run ran on; a **pre-dispatch guard** in `_run_via_worker` (and before spawn in the one-shot path) then finalizes as cancelled without running the job, and `_finalize_run` **never un-cancels** (an already-`cancelled` row / a recorded intent wins over a race-completed `ok=True`). The intent is only recorded for a run that's actually **in-flight** ‚Äî `start_execution` adds its id to `_inflight` at the top and drops it in the finally, and `stop_execution` gates the `_cancelled_ids.add` on membership (`_procs` alone is the wrong signal ‚Äî it's empty during a cold boot). That gate keeps a stop on an **already-finished** run ‚Äî e.g. the `run_sync` `?wait=true` timeout path, which calls `stop_execution` *after* the task ended ‚Äî from leaking an id into `_cancelled_ids` that nothing would ever discard. On Windows, cancellation uses `_terminate_proc_tree()` ‚Üí `taskkill /F /T` to kill the **whole child tree** (the `bash`/WSL/`python` subprocesses the agent spawned, each in its own process group), since a bare `proc.terminate()` would leave those orphans churning; POSIX signals the process only (the run shares the backend's process group ‚Äî no `setsid` ‚Äî and sandbox children self-reap on their own timeout). **Subprocess plumbing mirrors the engine exactly** (sync Popen + reader threads + `call_soon_threadsafe` + `CREATE_NEW_PROCESS_GROUP` + `_clean_env`) for the same Windows/debugpy reasons ‚Äî do not swap in `asyncio.create_subprocess_exec`. A worker's `asyncio.Queue` is bound to the loop that spawned it ‚Äî fine in production (one long-lived uvicorn loop); tests must drive multiple reuse runs inside a **single** `asyncio.run`.
- **Lifecycle**: `POST /scripts/{id}/preheat` eagerly spawns+preheats; the app lifespan preheats every `keep_warm` script on startup (fire-and-forget) and `shutdown_all`s on exit. The built-in assistant is always excluded (`worker_enabled` returns False). Frontend: a "Warm worker" section in the script Config panel (reuse / keep-warm toggles + Preheat button).

### WebSocket with replay buffer

`_WsManager` in `execution_engine.py` buffers every event per `execution_id`. WS clients connecting **after** a run started still get the full history via replay (`connect()` sends buffered then subscribes). Buffers are dropped 5 minutes after the run ends.

### Conversation threading (durable per-conversation agent state)

A `/converse` conversation IS a LangGraph **thread**: its `conversation_id` is the `thread_id`. The old chat contract was **stateless-per-turn** ‚Äî each turn was a fresh run getting only a `{role,content}` history slice, so the full LangGraph message state (tool results, a **read skill's body**) was discarded every turn and an agent **re-read its bound skill on every single turn**. Fixed by persisting the thread with a **durable SQLite checkpointer** (`langgraph-checkpoint-sqlite`, a baseline dep in **both** `venv_manager.BASELINE_PACKAGES` and `requirements.txt`).

- **Storage is per-script, process-independent**: `AsyncSqliteSaver` over `script_dir/workspace/threads.db` (WAL + `busy_timeout`), partitioned by `thread_id`. Because it's on disk (not in a process), it works identically for the one-shot runner, the warm worker, and across restarts ‚Äî a warm worker reuses one cached aiosqlite connection across jobs (`agentflow._THREAD_CHECKPOINTERS`, keyed by db path). Concurrent conversations of the same script share the file (WAL handles it); one conversation is serial. **Reclamation**: deleting a conversation (`routers/conversations.py::delete_conversation`) wipes its thread via `reset_thread`; deleting all of a conversation's messages resets the thread lazily on the next `chat_start` (anchor is None ‚Üí reset); deleting the script rmtrees the whole `workspace/` with it. The full thread is append-only and otherwise unbounded on disk (only the *model input* is trimmed, not the stored thread). **The one-shot runner MUST close these connections before it exits** (`agentflow._close_thread_checkpointers()`, called in the generated runner's `_main` `finally`): an `aiosqlite` connection is backed by a **non-daemon** worker thread, so a still-open checkpointer blocks interpreter shutdown ‚Äî the runner would emit its result then hang until `EXECUTION_TIMEOUT` kills it (and a pytest process hangs after the last test, which once wedged CI for the full 6-hour job cap). `tests/test_conversation_threads.py` closes them in an autouse teardown, and `.github/workflows/test.yml` sets `timeout-minutes` as a backstop. The **warm worker deliberately does NOT close between jobs** ‚Äî reusing one connection across jobs is the point; its process is terminated on shutdown/idle-reap.
- **Zero new "is-threaded" flag.** The engine promotes `input_data.thread_id` / `checkpoint_id` (set by `chat_start`) into `AGENTFLOW_THREAD_ID` / `AGENTFLOW_THREAD_CHECKPOINT` env (in `llm_envs`, so both the one-shot `_runner.py` and the worker `job_env` get them). `get_agent()`/`get_deep_agent()` **auto-attach** the checkpointer when `AGENTFLOW_THREAD_ID` is set (workspace present); `checkpointer=False` opts out, a saver instance overrides. Nothing set ‚Üí byte-for-byte the old stateless path. **Non-agent chat scripts (driving `get_llm()` directly) can't use a LangGraph checkpointer, so they keep using `input["history"]`** ‚Äî that's why `chat_start` still sends the history slice.
- **Send only the new message.** `stream_agent(agent, messages, thread_id=‚Ä¶, checkpoint_id=‚Ä¶)` (defaults from env) builds the run config and, when threaded, **reduces `messages` to the current turn** (`_latest_turn`: from the last human message onward) so a script that still prepends `input["history"]` doesn't double-count ‚Äî the checkpointer supplies the rest. Threading applies only when a thread id is in effect AND the agent has a checkpointer. `stream_agent_sync` bridges to the async path (`run_until_complete`, nest_asyncio) because the sqlite saver is async-only.
- **Context growth is bounded** by a `create_react_agent` `pre_model_hook` (`_make_trim_hook`) that trims the **model input** to a recent-token window (`AGENTFLOW_THREAD_MAX_TOKENS`, default 24000) via `trim_messages(strategy="last", include_system=True)`. It returns `llm_input_messages` (ephemeral) so the **full thread stays persisted** ‚Äî nothing is permanently dropped; a later turn re-trims from the complete history. The system prompt (which holds the skill menu) is always kept. **When threading is active `get_agent` uses a STATIC system-prompt string, not the classic callable `_prompt_fn`** ‚Äî a callable reads `state["messages"]` (untrimmed) and would defeat the trim.
- **Rollback = checkpoint anchoring, no separate endpoint.** `ConversationMessage.checkpoint_id` (Alembic **`0013`**) records the thread head after each assistant turn (read by `confirm_reply` via `services/conversation_threads.py::read_head_checkpoint`, a sync `SqliteSaver` over the same file). `chat_start` anchors the next turn at the **last surviving** assistant message's checkpoint (`input.checkpoint_id`); deleting the last turn in the UI makes the previous message's checkpoint the anchor ‚Üí the thread **forks back** to it (verified: resuming from an old checkpoint makes that lineage the new head). No surviving anchor ‚Üí `reset_thread` wipes the stale thread so a fresh one starts. So the existing delete-message frontend flow drives rollback with **no frontend change**. v1 boundary: arbitrary mid-history deletion isn't fully synced to the thread (the visible history loses it, the model thread keeps it); old pre-threading conversations start a fresh thread on their first threaded turn (no seed). Tests: `tests/test_conversation_threads.py`.

### Execution records: error persistence & retention

Two easy traps in `execution_engine.py`, both fixed ‚Äî keep them fixed:

- **Persist failures as logs, not just `execution.error`.** When a user script crashes, the runner emits a structured `{"type":"error", traceback}` event. The `_drain` loop now **also** writes it as an `ExecutionLog` (level `error`, step `error`) and pushes a `log` WS event ‚Äî otherwise the crash would only land in `execution.error` (shown as a transient toast) and be **invisible in the Logs/Output/Flow panels**, especially on reload. If the process exits non-zero but emitted *no* error (e.g. `sys.exit()`, SIGKILL/OOM, native crash), the engine **synthesizes** an error from the exit code so a failed run is never blank. `_mark_failed()` likewise persists a log.
- **Per-script execution retention.** `Script.max_executions` (default 50, `0` = unlimited; Alembic revision `0002`) caps how many execution *rows* a script keeps. `prune_executions(db, script_id, keep)` deletes the oldest **terminal** runs beyond `keep` (never in-flight ones) + their run dirs; it runs at the end of every run and on a `PATCH /scripts/{id}` that lowers the value. This is distinct from `_prune_old_runs(keep=20)`, which caps run *directories* on disk. Manual management: `DELETE /api/executions/{id}` (409 if the run is still in-flight ‚Äî stop it first) and `DELETE /api/executions?script_id=‚Ä¶` (clears all terminal records + dirs). Both wired into the script page's **Runs** tab (per-row trash + "Clear"), with the cap configurable in the Config panel ("Kept Executions").
- **Deleting a script reclaims its whole on-disk folder ‚Äî asynchronously.** `DELETE /api/scripts/{id}` (`routers/scripts.py::delete_script`) removes the DB row (SQLAlchemy `cascade="all, delete-orphan"` handles the files/executions/logs/cron/revisions/presets rows), then reclaims `data/scripts/<id>/` ‚Äî the per-script `.venv` (hundreds of MB), `runs/`, `workspace/`, and materialized files. Without freeing it the folder leaked forever on every delete (the retention pruning above is the *only* other thing that frees disk, and it never runs for an already-deleted script). The handler is `async` and first `stop_execution`s any in-flight runs (`running/queued/pending`) + `invalidate_worker`s so no live venv-python holds a Windows file lock during the rmtree. **The `rmtree` itself is moved OFF the request path** ‚Äî the venv is hundreds of MB and an inline delete made the endpoint slow; instead a FastAPI `BackgroundTask` (`_delete_script_dir_bg` ‚Üí `services/script_cleanup.py::delete_script_dir`, `ignore_errors=True`) frees the bytes *after* the 204 returns, so the UI updates immediately. **Fallback** for a background delete that never finished (crash, or a Windows lock leaving a partial folder): `services/script_cleanup.py::sweep_orphan_script_dirs(db)` `rmtree`s any `data/scripts/<dir>` whose name isn't a live `Script` id. It's scheduled on startup (in `app/main.py` lifespan, **after** `seed_assistant`) but **fire-and-forget in a worker thread** (`asyncio.create_task(_sweep_orphan_dirs())` ‚Üí `asyncio.to_thread`, own `SessionLocal`) so a stale hundreds-of-MB venv can neither slow startup nor block the event loop ‚Äî the no-orphan case is a cheap query + `iterdir` anyway. **Live scripts ‚Äî including the built-in AI assistant ‚Äî are protected because they still have a DB row**; only `<script_id>` folders live directly under `DATA_DIR` (skills/uploads/logs/db live one level up), so a dir with no matching row is unambiguously an orphan. Dot-dirs and stray files are skipped.

### Frontend ‚Üî Backend routing

Frontend is `output: "export"` (next.config.ts). `backend/app/main.py` has a catch-all that serves `frontend/out/`. Real routes are `/api/*` and `/ws/*`. The previous `frontend/src/app/api/[...path]/route.ts` proxy was removed ‚Äî don't reintroduce it; the static export can't have route handlers.

When you need `script_id` in the URL, use `?id=...` query (not path segment) ‚Äî that's the convention `/script` and `/converse` use because static export hates dynamic paths.

### External MCP tool injection

AgentFlow can connect to external MCP servers configured in the Tools UI. `script.mcp_server_ids` selects which enabled `MCPServerConfig` records are connected for a run. The runner builds `AGENTFLOW_MCP_CONFIGS`, creates a `MultiServerMCPClient`, injects tools into `agentflow._injected_tools`, and scripts access them through `get_tools()` / `get_agent()`.

The per-server connection dict is built by **one** helper ‚Äî `services/mcp_config.py::build_connection(srv, db)` ‚Äî used by both the runtime injector (`execution_engine`) and the "test connection" probe, so a probe reflects exactly what a run will see.

#### Test connection / tool listing (probe)

`POST /api/mcp-servers/{id}/probe` connects to a single server and returns `{ok, tools:[{name,title,description,input_schema}], error, needs_auth}`. It runs in the **backend** process via the raw `mcp` SDK (`services/mcp_probe.py`), inside a dedicated thread with a fresh `ProactorEventLoop` so `stdio` subprocess transport works regardless of the loop debugpy installs. The MCP client SDK (`mcp`) + `httpx` are backend deps now (`requirements.txt`); `langchain-mcp-adapters` stays a per-script venv baseline. The Tools UI exposes this as a per-server **Test** button that opens a dialog listing the tools + their JSON schemas.

#### OAuth 2.0 for remote MCP servers

Servers like Todoist / Fastmail sit behind OAuth ‚Äî a static `Authorization` header won't do. Because scripts run headless in subprocesses (no browser), the **backend** owns the flow (`services/mcp_oauth.py`):

- `MCPServerConfig.auth_type` = `none` | `oauth2`. `oauth_config` (JSON) caches discovered + manual endpoints and client creds; `oauth_token` (JSON) holds the live grant. **Neither is ever serialized to the frontend** ‚Äî `MCPServerOut` only exposes `auth_type`, a computed `oauth_connected`, and `oauth_scope`.
- `GET .../oauth/authorize-url` discovers the auth server (RFC 9728 ‚Üí RFC 8414 / OIDC), dynamically registers a client if needed (RFC 7591), generates PKCE, and returns a URL the UI opens in a popup. The `redirect_uri` is built from `settings.public_base_url` if set, else `request.base_url`. **Behind a reverse proxy you must set `PUBLIC_BASE_URL=https://host`** ‚Äî otherwise the redirect_uri is the proxy's internal http URL and providers reject it at the registration step (400). The Docker image runs uvicorn with `--proxy-headers` so a proxy that sends `X-Forwarded-Proto` also fixes the scheme; `docker-compose.traefik.yml` sets `PUBLIC_BASE_URL` from `$DOMAIN`.
- `GET .../oauth/callback` exchanges the code for tokens and renders a self-closing page that `postMessage`s `{source:"agentflow-oauth", ok}` back to the opener (Tools page listens and reloads).
- `POST .../oauth/disconnect` clears the token.
- At run time `build_connection()` calls `ensure_access_token()` (refresh-if-expired) and folds the bearer into `headers["Authorization"]`, so the subprocess only ever sees a static header ‚Äî no token objects crossing the `AGENTFLOW_MCP_CONFIGS` JSON boundary.
- `PATCH /api/mcp-servers/{id}` **shallow-merges** `oauth_config` so editing e.g. scope in the UI doesn't wipe the discovered endpoints / client_id.

### External secrets (credentials for user scripts)

Scripts must not hard-code API keys / tokens / webhook URLs. The `Secret` model (`secrets` table: `key`, `value`, `description`) holds them, managed via `routers/secrets.py` (admin-gated CRUD) and the `/secrets` page. Mirrors the channel/OAuth "secret never crosses to the frontend" contract:

- `SecretOut` reads `value` only to derive a computed `has_value` + masked `preview` (`Field(exclude=True)`); the raw value is **never serialized**. `SecretCreate.key` is constrained to `^[A-Za-z_][A-Za-z0-9_]*$` so the env-var mapping is unambiguous; the router also rejects keys that collide once upper-cased.
- At run time `execution_engine` loads all secrets and builds `AGENTFLOW_SECRET_<NORM(key)>` env vars (+ `AGENTFLOW_SECRET_NAMES`). These go into **`sub_env` only** ‚Äî deliberately **not** passed to `_write_runner()`, so secret values are never baked into the on-disk `_runner.py` (unlike `llm_envs`, which currently are). The diagnostic log prints secret **keys**, never values.
- Scripts read them via `agentflow.get_secret("<key>")` (case-insensitive, non-alnum ‚Üí `_`, same `_norm` as `get_llm`) / `list_secrets()`. Global by design ‚Äî single-admin model, every script sees every secret (no per-script opt-in like `mcp_server_ids`).
- **At rest the value is plaintext in the DB** ‚Äî consistent with `channels.api_key` and `oauth_token`; the DB lives on the protected data volume and is never exposed via the API. Encryption-at-rest would need a non-stdlib cipher (breaks `security.py`'s stdlib-only rule), so it's intentionally out of scope.

### Sandboxed exec tools (opt-in bash + python)

Two sandboxed exec capabilities live in **`backend/agentflow/_sandbox.py`** (stdlib-only, no new deps): `run_bash(command)` (a shell "exec sandbox") and `run_python(code)` (a full Python interpreter, e.g. for computation/data wrangling ‚Äî the final bare expression is echoed notebook-style). **They are OPT-IN ‚Äî deliberately NOT in the default `get_tools()`/`get_agent()`**, because they run arbitrary commands/code; this mirrors the `markdown()`/`image()` SDK-method model (the script enables them, they're not always on). A script uses them two ways:

- **Direct SDK calls** (like `markdown()`): `from agentflow import run_bash, run_python` ‚Üí returns `{stdout, stderr, returncode, timed_out}`.
- **As agent tools**: `bash_tool()` / `python_tool()` return LangChain tools named `bash` / `python`; `exec_tools()` returns both. Hand them to an agent explicitly: `get_agent(tools=get_tools() + exec_tools())` (or `+ [bash_tool()]` for bash only). Each factory takes optional `timeout=` / `cwd=` / `files=`.

**File access is opt-in per call** ‚Äî by default sandboxed code can't see the run's files (relative paths hit an empty throwaway dir; the `AGENTFLOW_*` path envs are scrubbed, so it can't even discover run_dir/workspace). Both `run_bash`/`run_python` and the tool factories accept: `files={"dest.csv": src_path_or_bytes}` (str = source path copied, bytes = content written; dest must stay inside the sandbox cwd ‚Äî traversal/absolute rejected; a non-dict `files` returns an error dict, never raises; originals untouched) and `cwd=<dir>` ‚Äî a **non-empty** dir (empty string / falsy falls back to a throwaway, matching the tool-description hint) e.g. `os.environ["AGENTFLOW_WORKSPACE_DIR"]`, run in as a caller-chosen **persistent** directory *not* deleted afterwards (sandboxed code can then modify it); the tool factories append the cwd to the tool description so the agent knows where relative paths land. `_run_sandboxed` **always** uses a separate throwaway `staging` dir for `run_python`'s `_harness.py`/`_user_code.py` (referenced by absolute path) distinct from the working dir ‚Äî so `files=` can never clash with harness files, even in the default case (the default working dir is its own throwaway, also `rmtree`'d).

**Isolation** (shared core `_run_sandboxed`, applied to both) is *process*-level, not a language jail ‚Äî the child can still touch the filesystem; the guarantees are: (1) **env scrub** ‚Äî the child keeps only a small allowlist (`PATH`/`HOME`/`LANG`/`LD_LIBRARY_PATH`/‚Ä¶), so **every `AGENTFLOW_*` var (secrets, LLM keys, OAuth tokens) is dropped** and sandboxed code can't read platform credentials; the allowlist **does** keep proxy / TLS config (`HTTP(S)_PROXY`/`NO_PROXY`/`ALL_PROXY` + lowercase, `REQUESTS_CA_BUNDLE`/`SSL_CERT_FILE`/`SSL_CERT_DIR`/`CURL_CA_BUNDLE`) ‚Äî since network is allowed by default, sandboxed code must reach the internet the same way the rest of the platform does; without it a **proxied** host silently "loses network" only inside the sandbox (direct connect ‚Üí `SSL: UNEXPECTED_EOF_WHILE_READING` / reset / timeout) while the main run (`_clean_env`, no proxy scrub) works ‚Äî proxy URLs are network config, not platform credentials; (2) **POSIX rlimits** via `preexec_fn` ‚Äî `RLIMIT_CPU` (busy-loop backstop), `RLIMIT_AS` (memory, default 1024 MB), `RLIMIT_FSIZE` (default 64 MB), `RLIMIT_CORE=0`, plus `os.setsid()` so a timeout kills the whole process group; (3) **wall-clock `timeout`** (default 30 s) ‚Üí SIGKILL the group; (4) **isolated temp cwd by default** (`tempfile.mkdtemp`, `shutil.rmtree`'d after; overridable per call via `cwd=`, see above). The Python sandbox runs under `python -I` (isolated mode) using **`sys.executable`** ‚Äî i.e. the per-script venv python ‚Äî so installed packages (numpy/pandas/‚Ä¶) are importable; bash runs via `bash -c` (falls back to `sh`). **The venv gets the same reach in bash**: `_sandbox_env()` prepends the interpreter's `bin`/`Scripts` dir (`os.path.dirname(sys.executable)`) to the sandbox `PATH`, so bash's `python`/`pip` and venv **console scripts** (e.g. a skill's CLI installed into the venv) resolve to that venv rather than a system/pyenv python ‚Äî without unscrubbing any env value. On **Windows** `_shell()` **prefers a real Git bash** (`_windows_git_bash()` probes the usual install paths / `git`-on-PATH) over the System32 `bash.exe` **WSL** launcher, because WSL is a separate Linux environment where the Windows venv is off-PATH and its `python.exe`/console scripts are unreachable (symptoms seen in the wild: `bad interpreter`, `command not found`, or a system `python3` with none of the venv's packages). `allow_network=False` best-effort blocks network via `unshare -rn` where the kernel permits it, silently falling back to allow ‚Äî **not** a hard guarantee; the real protection is the scrubbed env. Windows degrades gracefully (no `resource`/`preexec_fn`, keeps timeout + cwd + env-scrub + `CREATE_NEW_PROCESS_GROUP`). Non-configurable/global ‚Äî no DB model or UI; behaviour is code-only in the SDK.

### Cost / token observability (per-run LLM usage)

Every run records the LLM token usage it burned, so "which script is eating my API budget" is answerable. The tracer (`agentflow/_tracer.py`) already sees every LLM round-trip via `on_llm_end`; it now **accumulates** usage into a module-level tally (`_accumulate_usage` / `get_usage_totals`). `_extract_usage(response)` normalizes across provider shapes: prefers per-message `usage_metadata` (langchain-standard, ChatOpenAI/ChatAnthropic), falls back to `llm_output['token_usage']` (OpenAI) / `['usage']` (Anthropic). At the end of a run the generated runner emits **one** `{"type":"usage", prompt_tokens, completion_tokens, total_tokens, llm_calls}` event (in a `finally`, so partial usage before a crash still counts ‚Äî an unknown event type the drain loop otherwise ignores). `execution_engine._drain` stashes it and persists it onto the `Execution` row at finalization (both the completed and failed paths, before commit).

- **Columns**: `Execution.prompt_tokens / completion_tokens / total_tokens / llm_calls` (Integer, `server_default="0"`; Alembic **`0006`**, defensive add_column + inspector guard). Exposed on `ExecutionSummary`.
- **Dashboard endpoint**: `GET /api/executions/usage-stats?days=N` (`routers/executions.py::get_usage_stats`) aggregates overall totals + a **zero-filled daily trend** + a **per-script breakdown** (highest spend first). Bucketing is done **in Python** (not dialect-specific `date_trunc`) so sqlite + postgres both work; cheap because executions are retention-capped.
- **UI**: home page `UsagePanel` (last-7-day total + dependency-free CSS bar sparkline + top scripts) ‚Äî hidden until there's ‚â•1 run; per-run token badge in the script page's **Runs** tab (`compactNumber` in `lib/utils.ts`). Keep `_norm`-style provider handling generic. Tests: `tests/test_usage_stats.py`.

### Eval / regression testing (per-script test dataset)

Turns "did my prompt change help or hurt?" into a pass/fail number. A script has a **dataset** of `EvalCase`s (an input + assertions); an `EvalRun` executes every case and grades it, comparable across runs and pinnable to a `ScriptRevision` (the "improve ‚Üí eval ‚Üí don't regress ‚Üí promote" loop).

- **Reuse, don't reinvent**: each case runs through the **real** execution engine (`services/eval_engine.py::_run_case` creates an `Execution` + `await spawn_execution` + reads `output_data`), so it exercises the exact venv/LLM/tracer path a normal run does ‚Äî and every case is a first-class Execution with token usage tracked. The eval layer only orchestrates + grades.
- **Assertions** (`EvalCase.assertions` = JSON list of `{type, value, threshold?}`): `contains` / `not_contains` / `regex` / `equals` are pure string ops; `judge` is **LLM-as-judge** ‚Äî `_build_judge_llm` builds a model from the **default channel** by reusing `agentflow.get_llm` (no provider logic duplicated) via a transiently-set `AGENTFLOW_LLM_DEFAULT` in the backend process (the backend has the langchain stack, same as the assistant). The judge scores 0‚Äì10, passes at `>= threshold` (default 7), and is resilient to non-JSON replies.
- **Models**: `EvalCase` + `EvalRun` (Alembic **`0007`**, additive-table guard). `EvalRun.results_json` holds per-case detail (`{case_id,name,passed,output,error,execution_id,assertions:[‚Ä¶]}`), `passed`/`total` the headline, `revision_number` the version it ran against.
- **Router** `routers/evals.py` (`/api/evals`, admin-gated): cases CRUD (`/cases`), runs (`POST /runs` starts a background `start_eval_run` task, `GET /runs` list, `GET /runs/{id}` detail-with-results). A run is fire-and-forget; the frontend **polls** the run row for status (same pattern as executions).
- **UI**: `components/EvalPanel.tsx` ‚Äî a new **Eval tab** in the script page's bottom tabs (next to Runs). Dataset editor (per-case name + input JSON + assertions editor with type dropdown / value / judge threshold), **Run Eval** button, latest-result view with per-case pass/fail + failing-assertion detail + **delta vs previous run**, and a run history list. `lib/api.ts::evals`. Tests: `tests/test_eval_engine.py` (assertion grading + two end-to-end `run_eval`s through the real engine).

### Failure notifications (alert channels: PushPlus / Bark / email)

When a run reaches a **final** `failed` state, every enabled `NotificationChannel` gets pinged ‚Äî so a scheduled agent that dies at 3am isn't invisible. Deliberately generic (add a provider = one `_send_*` + a `_SENDERS` entry).

- **Model**: `NotificationChannel` (`notification_channels` table, Alembic **`0010`**) ‚Äî `name`, `type` (`pushplus` | `bark` | `email`), `enabled`, `config` (JSON, provider-specific, **contains secrets**). Same "secret never crosses to the frontend" contract as secrets/channels: `NotificationChannelOut` strips the secret sub-keys (`token` / `device_key` / `password`, in `schemas._NOTIFY_SECRET_KEYS`) and exposes only `config_safe` + a computed `has_secret`. The PATCH **preserves an existing secret** when the incoming sub-key is blank (the UI never re-sends stored secrets ‚Äî same "type to replace" pattern as the Tavily key / secrets).
- **Trigger tracking**: `Execution.trigger` (`manual` | `api` | `cron` | `rerun` | `eval`, default `manual`, same revision `0010`) is set at the 5 creation sites (`routers/executions.py` create/run_sync/rerun, `scheduler.py`, `eval_engine.py`). The notifier **skips `eval`** so a graded test case can't spam the channels.
- **Dispatch** (`services/notifications.py`): the engine calls `schedule_failure_notification(execution_id)` from `_finalize_run` (**only when `retry_count >= max_retries`** ‚Äî no alert on intermediate retries) and from `_mark_failed` (terminal engine failures). It's **fire-and-forget + best-effort**: a background task opens its own `SessionLocal`, and each send runs in `asyncio.to_thread` (sync httpx / smtplib) so a slow/broken webhook can't wedge run finalization or fail the run. De-duped via a bounded `_notified` set. Providers: PushPlus (`/send`), Bark (`POST {server}/{device_key}`), email (stdlib `smtplib`, STARTTLS or port-465 SSL).
- **Router** `routers/notifications.py` (`/api/notification-channels`, admin-gated): CRUD + `POST /{id}/test` (sends a canned message through the stored config, via `to_thread`, returns `{ok, error}`).
- **Frontend**: dedicated `/notifications` page (`app/notifications/page.tsx`, navbar Bell icon) ‚Äî list with enable toggle + Test/Edit/Delete, and a create/edit dialog with **per-type dynamic fields** (`FIELDS` map, labels via the `notifications` i18n namespace). `lib/api.ts::notificationChannels`. Tests: `tests/test_notifications.py` (masking, dispatch, eval/disabled/non-failed skips, + a full engine‚Üísend end-to-end) and two hook-fires-on-failure cases in `tests/test_execution_engine_run.py`.

### Web search provider (built-in web_search / web_fetch)

The built-in `web_search` / `web_fetch` tools (defined in `agentflow/__init__.py::_make_builtin_tools`) pick a provider from a **singleton config**, with **DuckDuckGo (via `ddgs`, no key) as the always-on fallback** so an unconfigured deployment still searches.

- **Model**: `SearchConfig` (singleton, `id="default"`; table `search_config`, Alembic `0003`) ‚Äî `provider` (`tavily` | `duckduckgo`) + `tavily_api_key`. Key follows the "never serialized to frontend" contract: `SearchConfigOut` exposes only `provider`, computed `tavily_connected`, and a masked `tavily_key_preview` (`Field(exclude=True)` on the raw key). Router `routers/search_config.py` (admin-gated): `GET`/`PUT /api/search-config` + `POST /api/search-config/test` (validates a Tavily key with a tiny live query, using a body key if given else the stored one).
- **Runtime**: `execution_engine` folds it into `AGENTFLOW_SEARCH_CONFIG` = `{provider, tavily_api_key?}` in **`secret_envs` (subprocess-only)** ‚Äî the Tavily key is *never* baked into the on-disk `_runner.py`, same as secrets. The diagnostic log prints the provider + whether a key is set, **never the key**.
- **Provider dispatch** (SDK helpers `_search_config`/`_tavily_search`/`_tavily_extract`/`_ddg_search`/`_httpx_fetch`): `web_search` ‚Üí Tavily `/search` when `provider==tavily` and a key is set, transparently falling back to DuckDuckGo on any Tavily error **or empty result**. `web_fetch` ‚Üí Tavily `/extract` (clean article text) when a key is set, falling back to httpx + BeautifulSoup (raw HTML if bs4 missing). Tavily auth is a `Authorization: Bearer <key>` header (httpx, no SDK/dep added). Adding a provider = add a helper + a branch here + a UI option; keep it generic (no per-vendor logic beyond search).
- **UI**: Tools page (`/tools`) "Web search provider" card ‚Äî provider select, masked Tavily key input (shows saved preview, type to replace), Save / Test / Remove-key. `lib/api.ts::searchConfig`.

### Database

- `app/database.py` switches on `DATABASE_URL` (sqlite vs anything else). SQLite gets WAL pragmas + `busy_timeout=5000` + `check_same_thread=False`. **Both sqlite (local/single-host) and postgres (docker) are first-class and must keep working.**
- **The connection pool is sized wide ON PURPOSE ‚Äî do NOT shrink it back to defaults.** sqlite: `pool_size=20, max_overflow=80` (=100); postgres: `20 + 40` (=60). This is a **correctness** requirement, not a perf tweak: the run endpoints are `async def` but do *synchronous* SQLAlchemy work on the event loop, and a long synchronous `POST /executions/run?wait=true` keeps its request coroutine (auth dep + engine session) holding connections for the whole run. With the SQLAlchemy default pool (5 + 10 = 15) a burst of concurrent runs momentarily needs >15 connections; the next checkout **blocks on the event-loop thread**, which starves the subprocess-drain callbacks (`loop.call_soon_threadsafe`) ‚Üí runs never finish ‚Üí connections never free ‚Üí **permanent deadlock** (reproduced live at exactly 10 concurrent `/run`; all hung indefinitely). A wide pool keeps checkouts non-blocking for realistic concurrency. Guarded by `tests/test_db_pool.py`. (Executions themselves are still bounded by `AGENTFLOW_MAX_CONCURRENT`, default 5 ‚Äî raise it for more run throughput; it is independent of the pool.)
- **Schema is owned entirely by Alembic** (`backend/alembic/`, driven by `backend/alembic.ini`). There is **no more `create_all` and no hand-rolled `V*.sql`/`schema_migrations` runner** ‚Äî that system was fragile (a drifted `schema_migrations` row re-ran an old `ALTER ADD COLUMN` and crashed startup with `DuplicateColumn`) and was removed. `env.py` reads the DB URL straight from `app.database` (single source of truth; also avoids configparser `%`-interpolation issues with postgres passwords) and sets `render_as_batch=True` on sqlite so `ALTER`s work there ‚Äî **the same revision files run on sqlite and postgres**.
- **Migrations auto-apply on startup**, so `docker compose pull && up -d` (or `F5`) self-migrates. `lifespan` calls `app/migrate.py::run_migrations(engine)`, which inspects the DB and picks one path (fail-fast: an error stops startup):
  - **`alembic_version` present** ‚Üí `alembic upgrade head`.
  - **no `alembic_version` but `scripts` present** ‚Üí a **pre-Alembic** deployment of *unknown / possibly partial* schema (old AgentFlow DBs used a fragile runner that could crash mid-way, leaving later columns/tables missing ‚Äî e.g. a DB that died on migration `V1` never got `skills`, `scripts.skill_ids`, etc.). We do **not** assume it matches any revision. Instead `_heal_schema_to_models(engine)` **reconciles it to the ORM models**: `create_all` builds missing *tables*, then each missing *column* is added **nullable + backfilled to the model default** (`col.type.compile(dialect)` for the type; a nullable ADD can't fail on a populated table; the backfill fixes API-required fields like `skill_ids`/`mcp_server_ids`/`max_executions` that must not be NULL). Then `command.stamp("head")`. **This heals ANY partial state** and is the real fix for a drifted cloud DB ‚Äî do not revert it to a naive `stamp("0001")` (that assumed a full baseline the crashed DB didn't have).
  - **empty DB** ‚Üí `upgrade head` builds everything.
- **Revisions** live in `backend/alembic/versions/`. `0001_baseline_schema` is the full schema **as a complete pre-retention deployment has it** (i.e. *without* later-added columns ‚Äî do not add new columns to it). `0002_add_max_executions` adds `scripts.max_executions` with a **plain `op.add_column`** (not `batch_alter_table` ‚Äî a pure add needs no sqlite table-recreate) and an **idempotent guard** (skips the ADD if the column already exists, since a DB that got it via the interim legacy migration `V11` would otherwise hit "duplicate column"). `0003_add_search_config` creates the singleton `search_config` table (web-search provider settings) with an **idempotent guard** (skips `create_table` if the table already exists, since a healed pre-Alembic DB may have gotten it via `create_all`). `0004_add_conversation_reasoning_effort` adds `conversations.reasoning_effort` (`String(16)`, server_default `off`) ‚Äî plain `op.add_column` + inspector guard, same defensive shape. Fresh/`alembic_version`-tracked DBs run these revisions; pre-Alembic DBs are healed to models + stamped `head` (revisions not replayed). **Write additive column migrations the same defensive way** (plain add_column + inspector guard); write additive *table* migrations with a `get_table_names()` guard.
- **When adding a column/table**: add the `Column`/model, then `cd backend && alembic revision --autogenerate -m "<desc>"`, review the generated file (autogenerate diffs models‚ÜîDB; `compare_type=True`), add a data backfill (`op.execute("UPDATE ‚Ä¶")`) if the new column is non-optional in the API schema, then just restart ‚Äî startup runs `upgrade head`. Give a column a `server_default` (mirrored on the model `Column`) so existing rows populate on ADD and autogenerate sees no drift. Use `op.batch_alter_table(...)` for column changes so sqlite works too. Do not drop the DB, and do not edit an already-shipped revision ‚Äî add a new one.
- To author/inspect out of band: `cd backend && alembic history`, `alembic current`, `alembic upgrade head`, `alembic upgrade head --sql` (preview DDL, e.g. `DATABASE_URL=postgresql+psycopg2://‚Ä¶ alembic upgrade head --sql` to eyeball the postgres SQL without a live DB).

### Skills (reusable agent instructions + files)

A **Skill** is an [Agent Skill](https://github.com/anthropics/skills): a folder with a `SKILL.md` (YAML frontmatter `name`/`description` + markdown instructions) plus any supporting files. Skills are **global** like MCP servers, and stored **purely on disk** under `backend/data/skills/<dir>/` (all of `backend/data/` is gitignored) ‚Äî *not* in the DB. This differs from scripts (whose files live in `script_files` rows): skills can bundle many/large files and marketplace install = drop a folder, so disk is the natural home. A script opts in via `script.skill_ids` (JSON array of skill **directory names**, AND-ed with each skill's `enabled` flag), exactly like `mcp_server_ids`.

- **On-disk layout & store**: `services/skill_store.py` is the *single* module that touches the skills dir. Each skill: `<dir>/SKILL.md` (`is_main`) + supporting files/nested folders + a `.agentflow.json` **sidecar** = `{enabled, source, installed_at, upstream, migrated_from_id}`. The **directory name is the stable identity** (used in `skill_ids` and `/api/skills/{id}` routes). Display **name/description come from the SKILL.md frontmatter** (parsed by a tiny dependency-free parser ‚Äî no PyYAML); AgentFlow-specific state lives only in the sidecar, so the rest of the folder stays a clean, portable Agent Skill. `skill_store` exposes `list_skills`/`get_skill`(returns `files` **and** `dirs`, incl. empty ones)/`create_skill`/`update_skill`(rewrites frontmatter for name/desc, sidecar for enabled)/`delete_skill`/`upsert_file`/`delete_file`/`create_dir`/`delete_dir`(rmtree a subfolder; skill root + `.agentflow.*` metadata protected)/`import_skill_dir`/`create_from_files`/`manifest_entry`. Path safety reuses `normalize_script_filename` (permits nested `a/b.md`).
- **Management**: `routers/skills.py` is a thin wrapper over `skill_store` (admin-gated; `{skill_id}` = dir name). Response shapes are unchanged (`SkillDetail` now also carries `dirs: list[str]`), so the frontend is untouched apart from folder support. New `POST /skills/{id}/dirs {path}` makes a **real on-disk folder** and `DELETE /skills/{id}/dirs/{path}` removes one ‚Äî `FileTree`'s "New folder" now persists empty dirs (via `onNewFolder`/`emptyDirs` props) and dir rows/context-menu offer **Delete folder** (via `onDeleteDir`). Scripts have no dir endpoint: they still chain folder‚Üífirst-file, and `onDeleteDir` there just deletes every file under the (virtual) prefix so the folder vanishes from the tree. The Tools page (`/tools`) lists skills; editing opens `/skill?id=‚Ä¶` reusing `FileTree` + `ScriptEditor` (Monaco). `FileTree` still supports collapsible tree + folder-preserving upload (`webkitdirectory` / `webkitGetAsEntry`). Pass `showRequirements={false}` for skills.
- **DB‚Üídisk migration**: `services/skill_migrate.py::migrate_skills_to_disk(db)` runs in the `app/main.py` lifespan (after `run_migrations`, mirroring `llm_migrate`). It writes any legacy `skills`/`skill_files` rows to disk (sidecar `migrated_from_id`) and rewrites every `script.skill_ids` entry from the old DB id ‚Üí new dir name. Idempotent; legacy rows are **kept** (read-only rollback backup). The `Skill`/`SkillFile` models stay defined but are read *only* by this migration.
- **Runtime ‚Äî two ways to consume skills** (`execution_engine` **copies** each enabled bound skill's folder `backend/data/skills/<dir>/` ‚Üí `run_dir/skills/<safe>/` via `shutil.copytree`, so both agent modes see skills under `run_dir` and can't mutate the stored copy; then builds the `AGENTFLOW_SKILLS` manifest `[{name, description, dir, main}]`; `AGENTFLOW_RUN_DIR` points at the run dir):
  - `get_agent()` (default, lightweight, progressive disclosure √Ý la deepagents): folds each skill's **name+description into the system prompt** and adds a built-in **`read_skill(name)`** tool that returns a skill's full `SKILL.md`. Built on `create_react_agent`; the agent can only read SKILL.md (supporting files reachable from script code via `skill_path`). No extra dependency.
  - `get_deep_agent()` (opt-in): builds a **deepagents** Deep Agent (`create_deep_agent`) whose `FilesystemBackend(root_dir=run_dir, virtual_mode=True)` mounts `skills/` at the **virtual** source `"/skills"`, so the agent browses+reads **every** skill file itself plus gets planning/sub-agents. **`virtual_mode=True` is mandatory on Windows** ‚Äî deepagents' path validator rejects drive-letter absolute paths (`D:\‚Ä¶\SKILL.md`), so a non-virtual backend crashes the instant the skills loader hands the agent a skill-file path (`Windows absolute paths are not supported`). Virtual POSIX paths (`/skills/‚Ä¶`) behave identically on every OS. **Skills are owned by deepagents' native `SkillsMiddleware`, NOT by our `read_skill` tool**: passing `skills=["/skills"]` (with the mounted backend) makes deepagents inject the bound-skill list + progressive-disclosure "`read_file` the SKILL.md path (`limit=1000`)" guidance into the system prompt every turn. So when native skills are active `get_deep_agent()` deliberately **does NOT add our `read_skill` tool or `_skill_preamble`** ‚Äî doing so used to hand the model a *second, conflicting* way to read a skill (read_skill-by-name vs read_file-by-path), and since our `read_skill` returns the whole SKILL.md as one blob, deepagents offloaded it to `/large_tool_results` and made the model `read_file` it back ‚Äî a clunky double-read that looked like the agent "kept re-reading the skill". `read_skill`/preamble remain only as a **fallback** for a deepagents build without `skills=`/backend support (the `native_skills` gate in `get_deep_agent`). Accepts a LangChain model instance (from `get_llm()`); introspects `create_deep_agent`'s signature across versions. Needs the `deepagents` baseline package.
  - The agentflow SDK skill functions (`list_skills`/`get_skill`/`skill_path`/`_make_skill_tool`/`_skills_root`) were **unchanged** ‚Äî they already read from the `AGENTFLOW_SKILLS` manifest + `run_dir/skills` on disk.
- Scripts can also read skills directly via `list_skills()` / `get_skill(name)` / `skill_path(name)`.

#### Skill marketplace (browse + install from official + community)

`routers/marketplace.py` (admin-gated, `/api/marketplace`) lets an admin browse and one-click install skills. **Key insight: both sources are GitHub-backed** (registries only *discover*; files always live on GitHub), so there is one install engine:

- `services/marketplace.py` ‚Äî the GitHub engine: `parse_github_ref` (owner/repo, `@ref`, tree/blob URLs, subpaths), `fetch_repo` (downloads a repo **tarball** once to `backend/data/marketplace/cache/`, extracts with stdlib `tarfile` + a path-traversal guard ‚Äî one request per source refresh, so we stay under GitHub rate limits), `scan_skills` (finds every `SKILL.md`), `install_skill` (copies the folder into the store via `skill_store.import_skill_dir`, sidecar `upstream` = `owner/repo[@ref]#subpath` for the installed-flag/idempotency), `official_catalog` (browses `anthropics/skills`). Optional `GITHUB_TOKEN`/`GH_TOKEN` env raises the rate limit.
- `services/marketplace_registry.py` ‚Äî **two** community search providers (discovery only; both return `githubUrl` per result ‚Üí install flows through the GitHub engine above):
  - **SkillsMP** (`search`): `GET https://skillsmp.com/api/v1/skills/search`, anonymous works (50/day), optional `SKILLSMP_API_KEY` Bearer raises it (‚Üí500/day), surfaces `X-RateLimit-Daily-Remaining`.
  - **skills.sh** (Vercel Labs, `search_skillsh` + `skillsh_token`): `GET https://skills.sh/api/v1/skills/search` **requires a bearer token** (`SKILLS_SH_TOKEN` or `VERCEL_OIDC_TOKEN`) ‚Äî anonymous requests 401, so without a token `search_skillsh` short-circuits to `{auth_required: true}` (no network call) and the UI shows a hint. Results map `source`‚Üí`githubUrl` (only GitHub-backed skills are kept).
- Endpoints: `GET /official?refresh=`, `GET /registry/search?q=&page=&sort=&provider=skillsmp|skillssh` (a needs-auth registry returns `{auth_required:true, skills:[]}` instead of erroring), `POST /install {owner?,repo?,ref?,subpath?,githubUrl?}` (a multi-skill repo returns `{needs_choice, skills:[‚Ä¶]}` for the caller to pick a `subpath`), `GET /sources` (returns `official` + a `registries[]` array).
- Frontend: `components/SkillMarketplaceDialog.tsx` (four tabs: Official anthropics/skills ¬∑ **From URL** ¬∑ Community SkillsMP ¬∑ Community skills.sh; cards with ‚≠ê + Install, `needs_choice` picker; native-`overflow-y-auto` scroll inside a `flex`-bounded `DialogContent`), opened from a **"Browse Marketplace"** button in the `/tools` Skills section; installs reload the skill list. The **From URL** tab (`installFromUrl`) takes a raw git/GitHub URL + optional skill name and calls `POST /install {githubUrl, skill}`, mirroring `npx skills add <url> --skill <name>` ‚Äî the backend `install` router filters resolved targets through `marketplace.pick_target(targets, name)` (matches the frontmatter name / folder basename / full subpath) so one named skill installs directly, else falls back to the `needs_choice` picker.
- **Out of scope** (noted, not built): auto-installing a skill's own Python deps or executing bundled skill scripts (no per-skill venv); a custom-GitHub-sources management screen (the engine accepts any `githubUrl`, but there's no sources-list CRUD UI).

### Reusable code modules (importable libraries, opt-in like skills)

A **module** lets scripts share Python *code* (unlike a skill, which shares prompt instructions/files). It's just a `Script` with **`kind="module"`** ‚Äî reusing the whole file-editor / FileTree / revisions stack for free (`Script.kind`, `module_package`, `module_ids`; Alembic **`0012`**; all module logic in **`services/module_support.py`**). A module has **no venv and is never run**; a script opts in via `script.module_ids` (row ids, AND-mirrors `skill_ids`/`mcp_server_ids`), then imports it as a package: `from <module_package> import ‚Ä¶`.

- **Identity & storage**: modules live in the same `scripts`/`script_files` DB rows as scripts (module code is structured records that need file CRUD, unlike skills' disk folders). `kind` discriminates; `module_package` is the importable package name (a valid Python identifier, **unique among modules**, defaults to a slug of the name). `list_scripts(kind=)` filters: `?kind=module` lists modules, anything else lists runnable scripts ‚Äî so **modules never appear in the run dashboard / Chat picker**. Created via `POST /scripts {kind:"module"}` (default file is `__init__.py`, not `main.py`). Editor is a dedicated **`frontend/src/app/module/page.tsx`** (mirrors the skill editor: FileTree + Monaco + requirements + package-name field + a "used by" panel from `GET /scripts/{id}/dependents`).
- **Import seam** (`module_support.materialize_modules`): at run time the engine copies each bound module's files into the **referencing** script's `script_dir/modules/<package>/` (skip-if-identical, the same truncate-race guard as script files; auto-creates `__init__.py` if the author didn't), then both runners **append `script_dir/modules` to `sys.path`** (`execution_engine.py::_write_runner` + `worker_pool.py::_worker_runner_source`) ‚Äî appended (lowest priority) so a module package never shadows the script's own files or an installed dep. Materialization runs **before** the warm-worker branch so a worker boots with the modules already on disk; the eager-preheat paths (`/scripts/{id}/preheat` + the startup `_preheat_keep_warm`) call `module_support.prepare_worker_dir` (writes script files + modules) before `acquire()` for the same reason (this also fixes the latent "first-ever preheat before any run has no main.py on disk" case).
- **Dependencies land in the *referencing* script's venv**: `module_support.effective_requirements(db, script)` = the script's own `requirements` **plus** every bound module's, line-deduped. All **three** install sites use it (`routers/scripts.py::install_deps`, `routers/ws.py` install action, `mcp_gateway.py::setup_script_env`) ‚Äî a module's deps are installed where the code that imports it runs. NB: **binding a module or bumping its deps requires re-running the referencing script's venv install** (no auto-reinstall of dependents ‚Äî surfaced, not automated).
- **Invalidation fan-out** (`module_support.dependent_script_ids` ‚Äî scanned live in Python since JSON-contains isn't portable): editing a module's **file** (PUT/DELETE) or its `requirements`/`module_package` (PATCH) retires **every dependent script's warm worker** (they imported the stale module at boot) and background-refreshes their cached input schema ‚Äî via `routers/scripts.py::_fanout_module_change`. Changing a referencing script's own `module_ids` (PATCH) retires **its** worker. (Without this fan-out a module edit would only retire the module's own ‚Äî never-booted ‚Äî worker, leaving dependents running stale code until idle-TTL reap.)
- **v1 boundaries** (documented tradeoffs): modules **can't bind other modules** (flat, no cycles ‚Äî a module's `module_ids` is ignored). Module-level globals persist across a warm worker's jobs (same serverless tradeoff as a script's own globals). Tests: `tests/test_module_support.py` (helpers) + `tests/test_execution_engine_run.py::test_script_imports_bound_module` (end-to-end import).

### Resource picker (unified skills/MCP/modules binding UI)

The script Config panel binds three opt-in resource kinds ‚Äî MCP tool servers, skills, and modules. They share **one** searchable picker (`frontend/src/components/ResourcePicker.tsx`) instead of three flat chip grids (which got messy at scale): the panel shows only the **selected** items as removable chips + a count summary + a **Manage** button; the dialog is a single search box + type filter (All/tools/skills/modules) over every available resource, selected-first. `script/page.tsx` builds a unified `ResourceItem[]` from `availableMcpServers`/`availableSkills`/`availableModules` and a `{mcp,skill,module}` selection map; `onChange` writes back the three `selected*Ids` arrays saved via the single `scripts.update` PATCH (`mcp_server_ids`/`skill_ids`/`module_ids`). i18n under `script:config.resources.*`.

### Authentication & API keys

The whole management UI/API sits behind a **single admin login**; external systems call the run endpoint with **issued API keys**. There is no multi-user model ‚Äî auth is a gate, not tenancy.

- **Crypto lives in one place**: `app/security.py` (stdlib only ‚Äî no new deps). PBKDF2-SHA256 password hashing; admin sessions are stateless HMAC-signed `<payload>.<sig>` tokens (JWT-ish); API keys are random `af_‚Ä¶` tokens of which only the SHA-256 hash is stored. The signing secret comes from `SECRET_KEY` or is generated once into `data/.secret_key` (persisted via the data volume).
- **Dependencies** are in `app/auth_deps.py`: `require_admin` (session cookie **or** `Authorization: Bearer <token>`) and `require_api_key_or_admin` (admin session **or** `X-API-Key` / `Authorization: Bearer af_‚Ä¶`). `current_subject()` is the non-raising variant for `/auth/status`.
- **Session transport is a cookie** (`af_session`, httpOnly, SameSite=Lax, `Secure` when `COOKIE_SECURE=true`). Cookies auto-attach to `fetch`, `<img>`/file downloads **and the WebSocket handshake** on the same origin ‚Äî that's why `<img>` artifacts and `/ws/*` work without manual token plumbing. WS handlers validate the cookie via `_ws_authenticated()` and close with code `4401` if invalid.
- **Wiring (`app/main.py`)**: `/api/auth/*` is public; every management router is included with `dependencies=[Depends(require_admin)]`. The **executions router is the exception** ‚Äî it gates per-endpoint so `POST /api/executions/run` can use `require_api_key_or_admin` while the rest require admin. `/health` and the frontend catch-all stay public so the login page can load.
- **Routers**: `routers/auth.py` (status/setup/login/logout/change-password/me) and `routers/api_keys.py` (list / create-once / revoke). `setup` only works when no admin exists yet (first-run).
- **Frontend gate**: `components/AuthGate.tsx` wraps the app in `layout.tsx`; it calls `/api/auth/status` on every navigation and routes to `/setup` (no admin), `/login` (not authed), or through. Protected pages don't mount until authorized. `lib/api.ts::req()` bounces to `/login` on a 401 (except for `/auth/*` calls). Pages: `app/login`, `app/setup`, `app/security` (account + API key management).
- **External API usage**: `curl -X POST http://host/api/executions/run -H "X-API-Key: af_‚Ä¶" -H "Content-Type: application/json" -d '{"script_id":"‚Ä¶","input_data":{‚Ä¶}}'` ‚Äî blocks until the script finishes and returns `{id,status,output_data,error,‚Ä¶}`.
- **Async run + completion webhook**: `POST /api/executions/run?wait=false` returns immediately with `{id,status:"queued",‚Ä¶}` (fire-and-forget); the run continues in the background. Pass `callback_url` in the body to be **POSTed** the final result on any terminal state (completed/failed/cancelled, after retries). Delivery lives in **`services/callbacks.py`** and is a **byte-for-byte structural clone of `services/notifications.py`** ‚Äî `schedule_completion_callback(execution_id)` fire-and-forget from the same two engine terminal paths (`_finalize_run` when NOT retrying + `_mark_failed`), own `SessionLocal`, sync `httpx.post` off the loop via `asyncio.to_thread`, bounded `_delivered` de-dup, a few retries then give up, and it **never raises into the engine** (a dead webhook can't wedge/fail a run). Unlike failure-notifications it fires on EVERY terminal status (the caller asked "tell me when it's done"), gated purely on `callback_url` being set. `Execution.callback_url` (Text, Alembic **`0011`**, plain add_column + inspector guard). To support polling, **`GET /api/executions/{id}` is now `require_api_key_or_admin`** (was admin-only) ‚Äî same exposure the key already has via `/run` + the `/mcp` gateway; the single-admin model has no per-run ownership to scope to.
- **Runs list search/filter**: `GET /api/executions` (`routers/executions.py::list_executions`, admin) takes `status` / `trigger` (comma-separated `in_` filters) + `q` (free-text). `q` is a **cross-run log search**: it matches the run id, its `error`, OR any of its `ExecutionLog.message`s (via an `id.in_(subquery)` so no join fan-out), case-insensitive `ilike` (works on sqlite + postgres). Frontend: the script page **Runs** tab (`RunsTab` in `app/script/page.tsx`) has a debounced search box + status filter chips; `executions.list(scriptId, {status,trigger,q,limit})` in `lib/api.ts`. Tests: `tests/test_execution_search.py`.
- **Drift guards** (`tests/test_dependency_sync.py`): two static asserts for the documented "keep in sync by hand" pairs ‚Äî every `venv_manager.BASELINE_PACKAGES` entry must appear in `requirements.txt`, and the two hand-copied `_norm()` bodies (`agentflow/__init__.py` ‚Üî `execution_engine.py`) must stay identical. They fail CI the moment either drifts.

### Outward MCP gateway (external coding agents develop scripts here)

AgentFlow **serves** an MCP endpoint (distinct from *consuming* external MCP servers above): Claude Code / Cursor / any MCP client connects to `POST /mcp` (Streamable HTTP, stateless, JSON responses) and gets tools covering the full write‚Üírun‚Üídebug loop ‚Äî `get_platform_context`, `get_scripting_guide`, script CRUD (`list/get/create/update_script`), file CRUD (`read/write/delete_script_file`, write returns ast-lint issues), `setup_script_env` (venv create + requirements install; slow first time), `run_script` (blocking, returns `output_data`/`error` **+ error logs/traceback** on failure), `list_executions`, `get_execution_logs`, and eval/regression (`list_eval_cases`, `add_eval_case`, `run_eval` ‚Äî so an agent can add graded test cases and run them). ### Time/timezone

Backend uses naive `datetime.utcnow()` everywhere (stored without TZ). Frontend `formatDate` / `toLocalDate` in `frontend/src/lib/utils.ts` append `Z` to TZ-less strings before parsing. **Don't change one side without the other**, or times will silently shift 8 hours.

**Cron scheduler timezone** is a *separate* axis (it only decides *when a crontab fires*, never how a timestamp is stored/shown ‚Äî those stay UTC). `services/scheduler.py::_resolve_timezone()` reads the `SCHEDULER_TIMEZONE` setting (an IANA name, `app/config.py`); blank ‚Üí APScheduler follows the process/container local zone (the standard `TZ` env), which is **UTC on the slim docker image unless set**. Both knobs need the tz database: `tzdata` is a pinned backend dep (`requirements.txt`) so `zoneinfo.ZoneInfo("Asia/Shanghai")` resolves even though `python:3.12-slim` ships no system tzdata and Linux `tzlocal` doesn't pull it in. A bad `SCHEDULER_TIMEZONE` never crashes startup ‚Äî it warns and falls back to local. `scheduler_service.effective_timezone()` exposes the resolved zone; `GET /api/cron-jobs/timezone` returns `{timezone, utc_offset, configured}` and the script-page Schedule tab shows a "Times are interpreted in ‚Ä¶" hint so users know which zone `0 9 * * *` means. `docker-compose.yml` documents both env knobs. Tests: `tests/test_scheduler.py`.

### Internationalization (i18n) ‚Äî English/Chinese UI

The frontend UI supports English + Chinese, switchable at runtime, on a **static-export** Next.js app (no server, no `[locale]` routing/middleware) ‚Äî so it's built as a plain client-side `i18next` + `react-i18next` + `i18next-browser-languagedetector` setup, not `next-intl`'s routing-based model.

- **All new user-facing text must go through `useTranslation`, never a raw string literal.** Add the key (English value = the literal text) to `frontend/src/lib/i18n/locales/en/<namespace>.json`, add the Chinese translation at the same key path in `locales/zh/<namespace>.json`, then `const { t } = useTranslation("<namespace>")` and render `t("key.path")`. Use `{{var}}` interpolation for dynamic text (`t("key", { count })`); use i18next's `_one`/`_other` suffixes for count-dependent plurals ‚Äî Chinese only ever needs `_other` (no plural distinction), English needs both.
- **One namespace per page or closely-related component group** (e.g. `dashboard`, `settings`, `tools`, `script`, `scriptEditor`, `scriptPanels`, `converse`, `assistant`, `common`) ‚Äî keeps translation JSON files small and lets unrelated pages be edited concurrently without touching a shared blob. `common` holds truly global chrome (app name, nav labels, the `ConfirmDialogProvider` default button labels, the language names themselves). Every namespace pair must be registered in `frontend/src/lib/i18n/config.ts`'s `resources` object (imports + the `en`/`zh` maps) ‚Äî a namespace that isn't registered there silently falls back to raw keys.
- **Language detection & switching**: **English is the product default.** Detection order is `localStorage` (key `agentflow_locale`) ‚Üí fallback `en` (`frontend/src/lib/i18n/config.ts`) ‚Äî browser `navigator.language` is deliberately NOT consulted, so a fresh browser always gets English; only the explicit choice made in Settings switches to Chinese. The user-facing switch lives in **Settings** (`components/LanguageSwitcher.tsx`, wired into `app/settings/page.tsx`'s "General" section) and calls `i18n.changeLanguage()`; the detector plugin persists the choice to `localStorage` automatically. Docs follow the same default: `README.md` is English, `README.zh-CN.md` is the Chinese mirror ‚Äî keep the two in sync when documenting new features.
- **`I18nProvider`** (`components/I18nProvider.tsx`) wraps the whole app in `app/layout.tsx`, alongside `AuthGate`/`ConfirmDialogProvider`/`AssistantProvider`.
- **Backend error messages are NOT translated server-side.** `HTTPException(detail=...)` strings must be stable, plain-English identifiers/sentences (a past inconsistency ‚Äî `routers/auth.py` mixed in Chinese while every other router used English ‚Äî was normalized away; don't reintroduce it). Translation happens **only** on the frontend: `frontend/src/lib/i18n/errorMessages.ts` maps a fixed set of known backend `detail` strings to Chinese, and `lib/api.ts`'s `req()` (via `extractErrorMessage()`) parses the JSON error body, looks up the translation, and falls back to the raw English string for anything not in the table. When you add a new stable `HTTPException` message that users will actually see in a toast, add its Chinese translation to `errorMessages.ts` too.

### Backend operational logging (loguru)

The FastAPI backend process has its own operational log ‚Äî startup/migrations, execution engine lifecycle, auth, MCP/OAuth, script/channel/secret CRUD ‚Äî via **loguru** (`backend/app/logging_config.py::setup_logging()`, called at the very top of `app/main.py` before uvicorn/sqlalchemy get a chance to configure stdlib `logging` themselves). It writes to the console (colorized) and a rotating file at `backend/data/logs/agentflow.log` (10 MB rotation, 14-day retention, zip compression). Level is controlled by the `LOG_LEVEL` env var (default `INFO`). Standard-library loggers used by uvicorn/sqlalchemy/apscheduler/the `mcp` SDK are intercepted and redirected into loguru so every line shares one format. `app/main.py` also has a request-logging middleware (method/path/status/duration; `/health` and `/_next/*` at DEBUG to avoid noise) and a global unhandled-exception handler (`logger.exception(...)` + a 500 response) ‚Äî an unhandled error is never silently swallowed.

**This is a separate concern from user-script execution logs** (the per-run `ExecutionLog` rows / WS log stream in `services/execution_engine.py`, fed by the runner's `__AGENTFLOW__`-prefixed stdout protocol) ‚Äî don't conflate the two. In particular, **never add loguru to `backend/agentflow/**`** (`_sandbox.py`, `_tracer.py`, `__init__.py`): that package is imported by user scripts running in their own subprocess/venv, and its `print()` calls are intentionally captured by the runner protocol as execution output shown in the Logs panel ‚Äî that's a different, already-working pipeline, not a bug to "fix" with a real logger.

- **Never log secret values**: API keys, LLM channel `api_key`, OAuth tokens/client secrets, and `Secret.value` must never appear in a log line ‚Äî log identifiers (id, name, key prefix) instead, matching the same rule already followed for the DB/API layer.

### Observability: Prometheus metrics (`GET /metrics`)

The backend exposes a Prometheus scrape target at **`GET /metrics`** (text exposition), all in **`services/metrics.py`** (backed by the `prometheus-client` dep ‚Äî backend-only, deliberately **NOT** in `venv_manager.BASELINE_PACKAGES`, since user scripts never emit metrics). This is platform-operational telemetry ‚Äî **distinct from** both the loguru operational log and the per-run `ExecutionLog` stream.

- **Two kinds of series, on purpose.** (a) **Process-lifetime counters/histograms** incremented at hook sites via `record_*` / `observe_*` helpers ‚Äî the rate/error/latency signals (a restart resets them; the DB still holds history). (b) **Point-in-time gauges** computed at scrape time by `_LiveCollector.collect()` from cheap **dialect-agnostic** DB counts (`count()`/`group_by`, works on sqlite + postgres, retention-capped so cheap) **and** live in-process state (the engine semaphore/`_procs`/`_tasks`, the WS managers, the warm-worker pool, the APScheduler). The collector is registered once into the default `REGISTRY`; `render()` returns `generate_latest()`.
- **Coverage.** HTTP (requests/latency by method + **matched-route template** ‚Äî bounded, never raw path); executions (throughput `agentflow_executions_total{status,trigger}`, duration, started, timeouts, retries, stops, pruned); **LLM spend** (`agentflow_llm_tokens_total{kind}` + `llm_calls`); the **profiler split** (`queue_wait`/`prep`/`cold_import`/`script` ‚Äî previously only in the `_prof` log line, now histograms); the warm-worker pool (boots/jobs/acquires/retirements + live count/alive/busy gauges); failure-notification & completion-webhook delivery; venv builds/installs; DB resource counts (scripts, executions-by-status/trigger, api-keys, channels, mcp-servers, cron-jobs, ‚Ä¶); and `agentflow_build_info` + config/feature-flag gauges.
- **Hard rules baked in** (see the module docstring): **never label by `script_id`/`execution_id`** (unbounded ‚Üí cardinality blowup; per-script spend already lives at `GET /api/executions/usage-stats`); only bounded labels (`status`√ó6, `trigger`√ó5, `kind`, `outcome`, `reason`, `preheat`, `ok`, ‚Ä¶). Every helper + every collector section is **exception-guarded** so instrumentation can never fail a run or a scrape, and one bad DB query can't blank the page. The module **degrades to a no-op** (helpers do nothing, `render()` returns a comment) if `prometheus-client` isn't installed. **Emit each metric family name once** ‚Äî a collector must yield one family with a series per instance, not one family per instance (a duplicate `HELP`/`TYPE` block is malformed exposition; `tests/test_metrics.py::test_no_duplicate_metric_families` guards this).
- **Terminal-counter hooks.** Each terminal execution is counted **exactly once** via `metrics.observe_execution(exc_row)`, called from every mutually-exclusive terminal path: `_finalize_run` (one-shot + worker completed/cancelled/failed), both timeout handlers, `_mark_failed`, `_mark_cancelled`. An auto-retry spawns a **new** Execution row that is counted again on its own terminal ‚Äî intentional (each attempt is a real run).
- **Single-process assumption.** The in-process gauges are one uvicorn worker's view (AgentFlow's normal deployment; the warm pool + WS replay buffers assume it too). Under `uvicorn --workers N` use the DB-backed gauges for a cluster-wide picture and prometheus multiprocess mode for the counters.
- **Auth (`app/auth_deps.require_metrics_access`).** Secure-by-default: needs an issued API key (`X-API-Key`/Bearer `af_‚Ä¶`) or an admin session, exactly like `POST /api/executions/run`. Two operator escape hatches for a trusted-network Prometheus: `AGENTFLOW_METRICS_PUBLIC=true` (open) or `AGENTFLOW_METRICS_TOKEN=<tok>` (a dedicated scrape credential accepted as Bearer / `X-Metrics-Token` header / `?token=` query). The endpoint is registered in `app/main.py` **before** the frontend catch-all; `/metrics` is added to `_QUIET_PREFIXES` so scrapes log at DEBUG. The HTTP middleware records `record_http(...)` for every request (except `/mcp`, which the gateway middleware short-circuits before routing).

## Conventions & contracts

- **User script entry point**: configurable per-script (`script.entry_function`, default `"run"`). Signature: `def run(input: dict) -> Any`. Return value goes into `execution.output_data`. A script can give `input` a typed contract with a module-level `INPUT_SCHEMA` (see *Script input schema*).
- **`get_llm()` resolution (channels)**: LLMs are configured as **channels** (NewAPI-style: one provider endpoint + key serving a list of models) in the `channels` table. `get_llm("<model-id>")` resolves a **model id** to `AGENTFLOW_LLM_<UPPER_ALNUM_>`; `get_llm()` reads `AGENTFLOW_LLM_DEFAULT`. `execution_engine` builds these envs by ranking enabled channels (`priority` desc, ties ‚Üí earliest `created_at`) and picking, per model id, the winning channel's creds ‚Äî so a model served by several channels uses the highest-priority one. The default is whichever channel set `is_default` + `default_model`. `_norm()` lives in both `agentflow/__init__.py` and `execution_engine.py` ‚Äî **keep them in sync**. The legacy `llm_configs` table is auto-folded into channels on startup (`services/llm_migrate.py`, idempotent) and otherwise unused; the `/api/llm-configs` router is legacy.
- **Provider mapping** in `agentflow.get_llm`: `anthropic` ‚Üí `ChatAnthropic`, `ollama` ‚Üí `ChatOllama`, `deepseek` ‚Üí `_ChatDeepSeekFixed` (subclass of `ChatDeepSeek` that patches `_get_request_payload` to echo `reasoning_content` back ‚Äî required for DeepSeek-R1 multi-turn), everything else falls through to `ChatOpenAI` with `base_url`. Default `timeout=60`, `max_retries=1` injected via `extra.setdefault`.
- **Reasoning / "think" level (`get_llm(reasoning=‚Ä¶)`)**: one shared level ‚Äî `None`/`"off"` | `"low"` | `"medium"` | `"high"` (also accepts `True`=medium) ‚Äî normalised by `_norm_reasoning` and mapped per provider by `_apply_reasoning`: `anthropic` ‚Üí `thinking={"type":"enabled","budget_tokens": ‚Ä¶}` (+ forces `temperature=1` and `max_tokens>budget`, both required by Claude); `openai` **without** a base_url (official o-series/gpt-5) ‚Üí `reasoning_effort=<level>`; `openai` **with** a base_url (compatible gateway: Qwen3/GLM/vLLM) ‚Üí `extra_body={"enable_thinking": true}` (boolean toggle ‚Äî level only gates on/off there); `deepseek` ‚Üí nothing (`deepseek-reasoner` reasons natively, text returns in `additional_kwargs["reasoning_content"]`); `ollama` ‚Üí `reasoning=True`. `get_agent`/`get_deep_agent` take `reasoning=` and forward it. **This is per-conversation user state, deliberately NOT a channel/global setting** ‚Äî see the chat convention below.
- **Chat page convention**: input is `{message, history: [{role, content}]}`, output is `{reply}` (with fallbacks: `message` / `response` / `result` / stringified). Maintained client-side in `frontend/src/app/converse/page.tsx` (Open-WebUI-style: collapsible searchable sidebar, flat full-width assistant messages, hover copy/delete, stop-generation, scroll-to-bottom). The chat page also consumes `trace` WS events and renders agent internals via `AgentNarrative` ‚Äî one **collapsed-by-default** "Agent trace" block above each assistant answer that expands to the readable story of the turn: the agent's intermediate text returns rendered as markdown, each tool call as a card with args/result/status (chronological, from `buildRows`). The last LLM turn's text is excluded (`excludeLastLlmText`) since the authoritative `content` renders it as the final answer. The shared markdown renderer is `components/Markdown.tsx`. On reload it re-hydrates from the run's `_trace` logs. (`AgentTraceInline.tsx` is the older compact-timeline view, no longer wired into the chat.) The tracer (`agentflow/_tracer.py`) is global, so any `get_agent()` / LangGraph run is traced automatically ‚Äî a plain `run()` that isn't a LangGraph node won't show node steps, but its tool/LLM calls still do. **Reasoning / chain-of-thought**: `splitThink()` in `converse/page.tsx` pulls the `<think>‚Ä¶</think>` (or `<thinking>`) reasoning out of the assistant `content` and renders it via a collapsible **`ThinkBlock`** ("Thinking‚Ä¶" while a tag is still open mid-stream, "Thought process" once closed) ‚Äî the answer is the content with the think block(s) removed. This is **required** because the shared `Markdown.tsx` (no `rehype-raw`) otherwise silently drops the unknown `<think>` HTML tag *and its contents*, making the reasoning invisible. `splitThink` is deliberately **tolerant of tag shape** ‚Äî it collects the contents of *every* `<think>` block (a script that naively wraps each streamed chunk as `<think>{chunk}</think>` is stitched into one block, not truncated to the first) and treats a trailing unclosed `<think>` as still-thinking ‚Äî so no script can break the render. **Emitting the think block is the platform's job, not the script's**: pass `stream_reasoning=True` to `get_llm`/`get_agent`/`get_deep_agent` and the SDK attaches `_ReasoningStreamer` (`agentflow/__init__.py`) ‚Äî a `run_inline` `BaseCallbackHandler` (like the tracer) that, on `on_llm_new_token`, wraps the model's separate chain-of-thought (`additional_kwargs["reasoning_content"]` for DeepSeek; Anthropic `thinking` content blocks) in a single `<think>‚Ä¶</think>` via `token()` (opens once, closes when the answer starts / at `on_llm_end`; a non-streamed `invoke()` gets a one-shot fallback from the final message). The script then contains **zero** think logic ‚Äî it just `token(chunk.content)` for the answer; reasoning never enters `reply` because the script never reads it. NB: LangChain calls `on_llm_new_token` with the token as a *keyword* `token=` in the sync path and positionally in others, so the handler takes `*args/**kwargs` (naming the param `token` would shadow the module's `token()` emitter). The manual `token("<think>‚Ä¶</think>")` pattern still works (and now renders robustly) but is legacy ‚Äî the flag is the documented path. **Delete button**: on `confirm`, the streamed assistant message's client temp id (`tmp-asst-‚Ä¶`) is swapped for the real DB id (and `animatingId` re-pointed) ‚Äî otherwise `canDelete` (gated on `!id.startsWith("tmp-")`) would hide the trash icon on the just-finished message until a reload. **Reasoning level is a per-conversation user choice, NOT a channel/global setting**: `ReasoningControl` (Brain icon, off/low/medium/high) writes `Conversation.reasoning_effort` (Alembic `0004`, `String(16)` server_default `off`); `chat_start` threads it into `input["reasoning"]`; a script forwards it via `get_llm(reasoning=input.get("reasoning"))` / `get_agent(reasoning=‚Ä¶)`. The chat templates (Streaming/Simple/Rich) already do this ‚Äî the Streaming one passes `stream_reasoning=True` so the platform surfaces the `<think>` block and the template's loop stays reasoning-free.
- **Baseline packages** auto-installed on venv create: see `BASELINE_PACKAGES` in `venv_manager.py`. Currently: `langchain-core`, `langchain-openai`, `langchain-deepseek`, `langgraph`, `langgraph-checkpoint-sqlite` (durable conversation threading ‚Äî persists a LangGraph thread to `workspace/threads.db`; pulls `aiosqlite`), `httpx`, `ddgs`, `beautifulsoup4` (web_fetch clean-text; falls back to raw HTML if absent), `langchain-mcp-adapters`, `nest-asyncio`, `deepagents` (powers `get_deep_agent()`). Users add more via `requirements.txt`. **Existing venvs don't get new baseline pkgs until recreated** ‚Äî but `web_search`/`web_fetch` *code* lives in the backend-imported `agentflow` SDK (not the venv), so provider changes apply to all scripts immediately; only the optional `beautifulsoup4` needs a venv rebuild. **This same stack is also listed in `backend/requirements.txt`** because the built-in assistant runs on the backend python rather than a per-script venv (see *Two Python runtimes*) ‚Äî when you change the langchain stack, update **both** `BASELINE_PACKAGES` (user scripts) **and** `requirements.txt` (backend + assistant).
- **MCP tool injection is per-script opt-in**: `script.mcp_server_ids` (JSON array of `MCPServerConfig.id`) controls which servers are connected at runtime. Empty = no MCP tools. The `enabled` flag on `MCPServerConfig` is a global availability switch (AND-ed with the per-script selection). Configure in the script's right panel.
- **Secrets & convenience helpers (`agentflow`)**: `get_secret("KEY")` / `list_secrets()` read externally-managed credentials (see *External secrets* above) ‚Äî keys are case-insensitive (`_norm`, same as `get_llm`). `http_get` / `http_post` / `http_request` are **provider-agnostic** thin `httpx` wrappers (default timeout / follow-redirects / raise-for-status) returning the `httpx.Response` ‚Äî they kill request boilerplate without locking the platform to any specific service. Add reusable cross-script helpers here rather than re-implementing them in each script; **keep them generic** ‚Äî no per-vendor logic (e.g. Bark/Slack/Todoist) belongs in the core SDK.

## Dev gotchas

- **`--reload` orphans running scripts**: editing backend code reloads uvicorn; running script subprocesses survive (process group detached) but their DB row sticks at `running` and WS dies. When testing scripts, either stop runs first or don't use `--reload`.
- **Frontend dev on :3000 can't hit `/api`**: `next dev` doesn't proxy. For full-stack dev either (a) hit `:8000` (use the static-built frontend), or (b) add a temporary rewrite in `next.config.ts` ‚Äî but remember `output: "export"` forbids rewrites in production builds.
- **Don't add LangSmith tracing** without thought ‚Äî `execution_engine` defaults `LANGCHAIN_TRACING_V2=false` / `LANGSMITH_TRACING=false` in sub_env to avoid surprise network calls and slow cold imports.
- **A run feels slow? Read the timing line on the backend console.** The engine never logged to stdout (only `_persist_log` ‚Üí DB/WS), so the uvicorn/F5 console stays blank during a run. `services/execution_engine.py::_prof` now prints one profile line per execution: `[agentflow] [<id8>] done status=‚Ä¶ rc=‚Ä¶ | queue_wait=‚Ä¶s prep=‚Ä¶s cold_import=‚Ä¶s script=‚Ä¶s total=‚Ä¶s`. It splits **queue_wait** (waiting for a concurrency slot) / **prep** (write files + DB queries + skill copytree) / **cold_import** (fresh `python` spawn + importing the langchain/langgraph/deepagents stack ‚Äî usually the dominant *fixed* cost when every run re-imports everything; the **opt-in warm worker pool** eliminates it on run #2+, see "Warm worker pool") / **script** (the user's `run()`, i.e. the actual LLM calls). The `cold_import`‚Üî`script` split is anchored by a `boot` event the generated runner emits right after `import agentflow` (an unknown event type the drain loop ignores), so it's accurate even for non-streaming scripts that only emit at the end. venv creation is **not** on this path (it's a separate `stream_create_venv` API run when the script is configured); a run only *selects* the venv python. Toggle the line off with `AGENTFLOW_PROFILE=0`.
